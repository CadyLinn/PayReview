# 花算 PayReview 個人財務決策 App

產品規劃版本：v0.2

## 一、品牌與產品名稱

### 產品名稱

花算 PayReview

### 名稱概念

「花算」取自「划算」的諧音，代表每一筆花費都值得先算一算。不只看見金額，也看見消費對預算與財務目標的影響。

PayReview 意指在付款前，快速檢視這筆消費是否符合使用者目前的財務計畫。

核心標語：

> 花錢前，先算價格與影響。

正式上架前，仍須完成商標、網域與 App Store 名稱檢索。

## 二、產品定位

花算 PayReview 不是單純的記帳 App，而是一個在消費發生前，幫助使用者看懂「這筆錢花下去會帶來什麼影響」的個人財務決策工具。

一般記帳軟體以事後紀錄為主，未必能實質改變使用者的消費習慣。PayReview 要解決的是：使用者在花錢前，不知道這筆消費會對今天的可用額度、近期生活支出與儲蓄目標造成什麼影響。

透過 PayReview，使用者可以在支出發生前，更明確理解這筆消費對財務狀況的影響，並更有意識地做出適合自己的財務決策。

### 產品承諾

在消費前，讓使用者知道：

1. 如果我現在花這筆錢，什麼會改變？
2. 我最近是否已經買得太頻繁？
3. 如果仍然想買，怎麼花費才能最少犧牲、影響原本的財務計畫？

### 目標使用者

有固定或規律收入、日常有彈性消費，且至少有一項想完成的儲蓄目標的年輕工作者。

使用者不想維護完整帳本，但希望在想花錢的當下得到清楚答案。

### 產品原則

1. 從事後紀錄轉為事前建議，在花費前理解消費對財務目標與預算的影響。
2. 不只顯示餘額，也點出每筆消費的機會成本。
3. 透過具體、可執行的建議，讓使用者安心決定每一筆消費。
4. 不以責備或羞恥機制影響使用者，而是清楚呈現選擇與代價。

## 三、產品模型

### 第一層：個人財務模型

內容：收入週期、帳戶與信用卡、必要支出、固定帳單、安全緩衝、財務目標與不可犧牲項目。

回答：什麼對這個人重要？

### 第二層：即時財務狀態

內容：已發生交易、待入帳交易、預期收入、即將到期帳單、已承諾支出與資料更新狀態。

回答：現在真實的財務情況是什麼？

### 第三層：決策引擎

內容：安心可花額度、今日／本週／下次收入日前建議範圍、消費速度、未來現金流、機會成本、目標延期、風險門檻與恢復方案。

回答：這個選擇會造成什麼？

### 第四層：導航體驗

內容：今天頁、Decision Card、消費前試算、交易後回饋與風險提醒。

回答：使用者現在需要知道什麼，以及可以採取什麼行動？

### 計算原則

1. 先預留必要與預期型支出，再呈現彈性可花金額。
2. 先預留優先保護目標的最低儲蓄額度。
3. 用於結論判定的保守可支用額度，必須取「剩餘彈性預算」與「扣除預期支出上限及受保護目標提撥後的安心可花下限」兩者中較低者。
4. 安心可花金額要呈現為範圍或估算，不做不合理的絕對承諾。
5. 每一個結論都能回看所使用的資料與假設。
6. 所有金額計算由可測試的規則引擎完成；AI 未來只負責將計算結果轉為容易理解的文字。

## 四、核心消費前體驗

PayReview 的消費前評估固定回答三件事：

1. 預算：買完後，今天、本週與下次收入日前的彈性額度剩多少？
2. 頻率：最近是否已經有多次相同類型的消費？
3. 目標：這筆消費是否影響財務目標？若有影響，如何恢復原本計畫？

當規劃外消費即將發生時，App 應提醒使用者這筆消費的機會成本與財務影響，但不能把使用者貼上負面標籤。

## 五、App 資訊架構

App 主要包含四個頁面：

### 1. 今天

回答：我現在最需要知道什麼？

內容：安心可花金額、下一筆預期支出、目標狀態與評估消費入口。

### 2. 評估

回答：如果我買這個，會發生什麼？

內容：快速輸入金額、預算／頻率／目標三種影響卡片，以及購買、延後或略過選項。

### 3. 計畫

回答：我已經承諾了哪些事？

內容：收入週期、預期支出、彈性預算、安全緩衝、目標與不可犧牲項目。

### 4. 紀錄

回答：最近發生了什麼？我做了哪些選擇？

內容：實際交易、延後購買、略過消費與計畫調整。

設定頁放在次要位置，包含登入、帳號、隱私、資料匯出與帳號刪除。通知設定待 P1 加入本機通知時再提供。

## 六、啟動與登入流程

### 啟動流程

1. 顯示 Icon 動畫，最多 0.8 秒。
2. 若已有有效 Firebase 登入狀態與 Firestore 財務計畫，直接進入「今天」。
3. 若尚未登入，進入歡迎與登入頁。
4. 登入成功但尚未建立財務計畫時，進入首次設定。

### 登入頁文案

標題：付錢前，先看影響。

說明：輸入眼前金額，也看見這筆錢會帶來什麼改變。

### 登入選項順序

1. 使用 Apple 繼續。
2. 使用 Google 繼續。

使用者必須使用 Apple 或 Google 帳號登入後才能進入首次設定與核心功能。不提供訪客模式或匿名登入。

MVP 將跨裝置財務計畫與紀錄連續性定義為重要的帳號功能，Cloud Firestore 是唯一正式資料來源，因此沒有可與雲端帳號分離的本機訪客資料來源。送審前必須依當時有效的 App Review Guideline 5.1.1(v) 再次確認此登入要求；若無法證明帳號對核心功能具有必要性，必須先調整登入方案，不得直接送審。

### 登入規則

1. iOS 首選 Sign in with Apple。
2. 若提供 Google 登入，也必須提供符合 Apple 規範的等效登入選項。
3. 登入時不要求通知權限、金融帳戶權限或行銷同意。
4. 設定頁必須提供帳號刪除與資料匯出。帳號刪除需先完成近期重新驗證；使用 Sign in with Apple 時，必須在刪除 Firebase Authentication 身分前撤銷 Apple 授權，並刪除 Firestore、雲端檔案、快取及本機敏感資料。
5. Firebase Authentication 是 MVP 唯一登入服務。

## 七、首次設定流程

目標：在 2 至 4 分鐘內完成。重點是先得到可用答案，而不是建立完整帳本。

### 第一步：一個重要目標

問題：你想讓這筆錢守住什麼，或前往什麼？

輸入：

1. 目標名稱與目的，例如日本旅遊。
2. 目標金額與目前已存金額。
3. 目標達成日期。
4. 優先程度：優先保護、重要、可調整。

若使用者暫時沒有目標，系統建立「安全緩衝」作為預設目標，之後可以更名。

### 第二步：收入週期與可支配金額

問題：你的錢通常什麼時候進來？

輸入：

1. 每月、每兩週、每週或自訂週期。
2. 下一次收入日。
3. 扣除必要扣款後，可自由安排的收入估計值。

不詢問公司、薪資明細與銀行帳戶。

### 第三步：預期型固定支出

問題：每個月有哪些固定支出一定需要支付？

快速模板：房租、交通、午餐、訂閱、帳單、其他。

支援每日與每月支出。每一筆可設定：

1. 預估金額或金額範圍。
2. 頻率或剩餘次數。
3. 預計日期或到期日。
4. 是否為必要或不可犧牲支出。

範例：午餐，約 NT$120，本週還有 3 次。

### 第四步：彈性預算

問題：你想為娛樂、興趣或固定支出超支等非固定支出，設定多少可用額度？

輸入：

1. 每週或每月週期。
2. 彈性預算金額。
3. 提醒語氣：溫和或直接。

### 規劃外支出判定

超過保守可支用額度的開銷應被視為規劃外或超出預算的支出，並觸發機會成本與財務影響提醒。即使帳面剩餘彈性預算較高，也不能使用已預留給預期支出或受保護目標的金額提高判定結果。

不能因超過彈性預算就直接稱為「衝動消費」，也不能直接宣告目標一定延期。只有在實際動用目標預留金，或使最低儲蓄速度無法維持時，才顯示目標延期。

### 完成頁

> 你的計畫準備好了。下次收入日前，扣除預期支出與目標後，約有 NT$680 可自由安排。

## 八、快速消費評估

進入頁面後，只先詢問金額，分類為選填。

畫面內容：

- 正在考慮花錢嗎？
- `NT$＿＿＿＿`
- 快速金額：50、100、200、500。
- 類別：宵夜、飲料、娛樂、購物、其他。
- 按鈕：查看影響。

結果頁先顯示一句清楚結論，再呈現細節。

範例：

> NT$600 可以花，但會壓縮這週的彈性空間。

- 本週彈性預算：NT$680 → NT$80。
- 日本旅遊：若使用彈性預算支付，目標日期不變。
- 若想保留原本彈性額度：未來 5 天平均每天少花 NT$120，即可平衡。

系統必須支援三種情境：

1. 從彈性預算支付：顯示剩餘額度與安心可花範圍。
2. 保護目標日期：顯示需要從其他地方減少多少消費。
3. 動用目標基金：只有使用者主動選擇時，才顯示目標可能延後多久。

## 九、今天頁

今天頁不是充滿圖表的 Dashboard，而是一個財務導航頁。

資訊順序：

1. 安心可花金額，例如「今天約可安心花 NT$520」，並說明已預留的支出與目標。
2. 今日預期支出，以類似待辦事項的方式呈現，可標記已花、今天跳過或修改預估。
3. 新增並評估一筆規劃外消費，例如宵夜或夜唱。這是頁面上唯一最明顯的主要行動，不能藏在加號或選單中。
4. 一個目標狀態，例如「日本旅遊，進度 42%，目前仍在軌道上」。
5. 一則有用觀察，例如「你本週的彈性消費速度比平常快」。

第一版不放一般記帳 App 常見的大量圓餅圖與報表。

## 十、日常記帳

提供一般記帳 App 的基本帳務紀錄、標籤分類與分析功能：

- 新增收入。
- 新增支出。
- 轉帳。
- 類別管理。
- 日期。
- 備註。
- 標籤。

轉帳不應計入收入或支出。

## 十一、計畫與紀錄頁

### 計畫頁

用來修改所有計算假設：

1. 收入週期。
2. 帳戶與信用卡；第一版為選填，只作為可見性資料。
3. 必要或不可犧牲支出。
4. 預期型支出與剩餘次數。
5. 彈性預算。
6. 安全緩衝。
7. 財務目標。
8. 使用者不願意犧牲的項目，例如健身、與家人聚餐或醫療需求。

### 紀錄頁

保留：

1. 實際交易。
2. 延後購買。
3. 略過的消費。
4. 計畫調整。
5. 資料來源：手動、分享文字、截圖或匯入檔案。

本週回顧可顯示：

- 固定支出花費。
- 規劃外消費金額。
- 儲蓄目標進度。

若一筆實際交易對應到某筆預期支出，必須將該預期支出標記為完成，不能重複扣除預算。

## 十二、初步技術方向

第一版採 iOS-first。

1. App：SwiftUI。
2. 決策引擎：獨立 Swift 模組，集中處理預算、目標與頻率計算，確保可測試且可離線使用。
3. 資料：Cloud Firestore，並啟用離線持久化。
4. 同步：Firebase Authentication + Cloud Firestore。
5. 登入：Sign in with Apple 與 Google Sign-In，使用者必須擇一登入。
6. P1 才加入 Share Extension、手機端 OCR 與本機通知。

### 主要資料模型

- `IncomeCycle`：收入週期。
- `PlannedExpense`：預期型支出。
- `FlexibleBudget`：彈性預算。
- `Goal`：儲蓄目標。
- `SpendScenario`：輸入金額後的消費試算。
- `Transaction`：正式交易。
- `DeferredPurchase`：晚點決定的消費。
- `DecisionSnapshot`：當下計算使用的依據與結果。

### 重要資料規則

1. 使用者選擇「購買並記錄」前，`SpendScenario` 永遠不是正式交易。
2. 使用者只進行試算時，不得改變真實預算。

### 隱私原則

1. 不連結銀行也能完整使用第一版。
2. 只有在功能真的需要時，才請求權限。
3. 分享文字與截圖盡可能在手機端解析。
4. 第一版即提供資料匯出與刪除。

## 十三、品牌與 Icon 方向

### 品牌個性

冷靜、實用、溫和鼓勵。像一位讓使用者看見代價的朋友，不像銀行後台，也不讓使用者因花錢而感到罪惡。

### Icon 概念

一枚硬幣變成一條向前的路徑。

設計元素：

1. 外圈代表金錢與個人財務邊界。
2. 圓圈刻意留一個缺口，缺口延伸成向前的箭頭或路徑。
3. 表示每筆錢都是一個選擇，也會改變前往目標的節奏。
4. 不使用貨幣符號、豬公、上升圖表或吉祥物，避免看起來像一般記帳 App。
5. Icon 不放文字，在小尺寸下也要容易辨識。

### 視覺方向

1. 深墨綠或深藍綠作為底色，建立信任與專注感。
2. 柔和薄荷綠或藍綠代表安全、可花與順利進行。
3. 暖琥珀色代表需要留意的取捨；除非真的有風險，否則不使用紅色。
4. 使用大金額、大留白，每個畫面只保留一個主要行動。

### 動畫方向

1. 開啟 App 時，硬幣外圈在約 0.8 秒內畫出，再打開成路徑。
2. 新手設定的進度以路徑慢慢前進呈現，不製造打卡或闖關感。
3. 消費評估結果依序出現：預算影響、消費頻率、目標影響與恢復方式。
4. 使用者確認購買後，小金額標記從「彈性額度」移到「已花費」，但不為花錢播放慶祝動畫。
5. 必須支援系統的「減少動態效果」設定。

## 十四、下一步待確認決策

1. PayReview 的正式商標、網域與 App Store 名稱檢索。
2. 第一版是否只服務台灣使用者，並預設新台幣與台灣日期格式。
3. 安心可花金額要以「下次收入日」為主，或同時提供「本月」視角。
4. 哪些支出模板預設為必要支出。
5. 是否強制使用者建立一個目標，或允許只建立安全緩衝。
6. 消費頻率預設採最近 30 天、收入週期，或允許使用者自訂。
